The Problem: Edge Inference Performance

You have a vision model. You want to run it on the edge. The vendor says "67 TOPS!" but what does that mean for your actual workload?

I ran YOLOv8n through the full optimization pipeline on a Jetson Orin Nano to find out. This post covers the journey from PyTorch to TensorRT, with actual numbers at each step.

The Optimization Pipeline

Step 1: PyTorch Baseline

Start with the Ultralytics YOLOv8n model. This is our baseline — no optimization, just PyTorch with CUDA. 

from ultralytics import YOLO

model = YOLO("yolov8n.pt")
results = model.predict(source="bus.jpg")

Step 2: Export to ONNX

ONNX is the intermediate representation. TensorRT consumes ONNX and produces an optimized engine. 

yolo export model=yolov8n.pt format=onnx imgsz=640

Step 3: Build TensorRT Engines

TensorRT performs layer fusion, kernel auto-tuning, and precision calibration. Each engine is hardware-specific — what's built on Orin Nano runs only on Orin Nano. 

# FP32 — baseline TensorRT
trtexec --onnx=yolov8n.onnx --saveEngine=yolov8n_fp32.engine

# FP16 — uses Tensor Cores
trtexec --onnx=yolov8n.onnx --fp16 --saveEngine=yolov8n_fp16.engine

# INT8 — requires calibration for production
trtexec --onnx=yolov8n.onnx --int8 --saveEngine=yolov8n_int8.engine
Note: INT8 without a calibration dataset uses simple min-max scaling. For production, provide representative images for better accuracy.

Step 4: Benchmark Each Engine

# Run benchmark with trtexec
trtexec --loadEngine=yolov8n_fp16.engine \
        --batch=1 \
        --warmUp=500 \
        --duration=10 \
        --avgRuns=100

Analysis

PyTorch → TensorRT FP32

Even without precision reduction, TensorRT provides significant speedup through kernel fusion and memory optimization. Convolution + BatchNorm + ReLU sequences become single fused kernels.

FP32 → FP16

FP16 enables Tensor Core utilization. The Orin Nano's 32 Tensor Cores are designed for mixed-precision workloads — FP16 compute with FP32 accumulation. Memory bandwidth is halved, and throughput roughly doubles.

FP16 → INT8

INT8 reduces memory footprint and can improve throughput further, but accuracy depends on calibration quality. For object detection, INT8 typically works well with minimal mAP loss.

Profiling with Nsight Systems

To understand why FP16 is faster, profile with Nsight Systems: 

nsys profile --trace=cuda,nvtx \
    trtexec --loadEngine=yolov8n_fp16.engine --batch=1 --duration=5

The trace shows Tensor Core kernels (names containing hmma or imma) dominating the GPU timeline, with minimal memory transfer overhead.

Practical Considerations

Memory Management

The Orin Nano has 8GB unified memory. Running the desktop environment consumes ~1-2GB. For maximum memory headroom: 

# Drop to multi-user (no GUI)
sudo systemctl isolate multi-user.target

# Clear filesystem cache
sudo sysctl -w vm.drop_caches=3

Thermal Throttling

Sustained inference can cause thermal throttling. Monitor with jtop and ensure adequate cooling. The stock heatsink works for burst inference; active cooling helps for continuous workloads.

Engine Portability

TensorRT engines are not portable across GPU architectures. An engine built on Orin Nano (Ampere) won't run on Xavier (Volta). Rebuild for each target.

Conclusion

Key takeaways:

  • TensorRT FP16 is the sweet spot — Tensor Core acceleration with no accuracy loss
  • INT8 offers additional gains but requires proper calibration
  • Power efficiency is excellent — real-time object detection at under 15W is achievable
  • Memory is the constraint — optimize for memory when deploying larger models
