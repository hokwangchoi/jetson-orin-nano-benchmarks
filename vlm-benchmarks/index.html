VLMs on Jetson Orin Nano: Fitting Vision-Language Models in 8GB

Hokwang Choi · April 2026 · 10 min read

TL;DR

Challenge: Run VLMs on 8GB unified memory
Models tested: Qwen2.5-VL-3B (INT4), Cosmos Reason 2B (FP8)
Key metrics: TTFT, tokens/sec, peak memory
Finding: Sub-4B models fit with aggressive quantization
Code: github.com/hokwangchoi/jetson-orin-nano-benchmarks

The Challenge: VLMs on Edge Hardware

Vision-Language Models are transforming what's possible with AI — they understand images and generate text. But they're memory-hungry. A typical 7B VLM needs 14GB+ in FP16, and we only have 8GB.

Can we run VLMs on the Jetson Orin Nano? Yes — but it requires careful model selection and aggressive quantization.

Memory Budget

The Orin Nano has 8GB unified memory shared between CPU and GPU:

Total RAM8192 MB
System/OS~800 MB
Desktop (GNOME)~1200 MB
Available for model~6000 MB

With the desktop running, we have ~6GB. Kill the GUI and we get ~7GB. That's our ceiling.

Model Selection

VLM memory is dominated by the LLM decoder. A 7B model in FP16 needs 14GB — way over budget. We need smaller models with quantization:

Model Parameters FP16 Size INT4 Size Fits?
LLaVA-1.5-7B 7B 14 GB ~4 GB Tight
Qwen2.5-VL-3B 3B 6 GB ~2 GB ✓
Cosmos Reason 2B 2B 4 GB ~1.5 GB ✓

I focused on Qwen2.5-VL-3B and Cosmos Reason 2B — both designed for efficiency while maintaining strong vision-language capabilities.

Benchmark Methodology

Metrics

  • TTFT — Time to first token. How long until generation starts?
  • Tokens/sec — Generation throughput
  • Peak memory — Maximum GPU memory during inference
  • Power — Average power draw

Test Setup

  • Input: 384×384 image + short prompt ("Describe this image in detail")
  • Output: 100 tokens generated
  • Warmup: 2 inference passes
  • Measured: 10 passes, averaged

Deployment Options

Option 1: Jetson AI Lab + llama.cpp

The Jetson AI Lab provides optimized containers. For VLMs, llama.cpp with GGUF models is the most memory-efficient option: 

# Pull the container
docker pull dustynv/llama-cpp:r36.4.0

# Run with GPU
docker run --runtime nvidia -it --rm \
    -v ~/models:/models \
    dustynv/llama-cpp:r36.4.0

# Inside container
./llama-cli -m /models/qwen2.5-vl-3b-q4.gguf \
    --image /path/to/image.jpg \
    -p "Describe this image"

Option 2: TensorRT-LLM

For maximum performance, TensorRT-LLM provides optimized inference. However, VLM support is still maturing: 

# Build engine (on Jetson)
python build.py --model_dir ./qwen2.5-vl-3b \
    --dtype float16 \
    --use_weight_only \
    --weight_only_precision int4
Warning: TensorRT-LLM build times on Jetson are long (1-2 hours). Consider cross-compiling on a workstation if iteration speed matters.

Option 3: ExLlamaV2

ExLlamaV2 offers excellent quantization with good performance on consumer hardware: 

from exllamav2 import ExLlamaV2, ExLlamaV2Config

config = ExLlamaV2Config()
config.model_dir = "./qwen2.5-vl-3b-exl2"
config.max_seq_len = 2048

model = ExLlamaV2(config)
model.load()

Memory Optimization Techniques

1. Kill the Desktop

# Switch to text mode
sudo systemctl isolate multi-user.target

# Back to desktop later
sudo systemctl isolate graphical.target

This frees ~1.2GB immediately.

2. Clear Caches

sudo sysctl -w vm.drop_caches=3

3. Use Flash Attention

If your framework supports it, Flash Attention reduces memory for long sequences. Critical for VLMs processing high-resolution images.

4. Streaming Decode

Generate tokens one at a time rather than batching. Trades throughput for lower peak memory — necessary when you're at the edge of capacity.

Use Case: Driving Scene Understanding

I tested with driving scene images from nuScenes. The prompt: 

"You are an autonomous vehicle perception system. 
Describe the scene: identify vehicles, pedestrians, 
traffic signs, and potential hazards."

Sample Output

The scene shows a four-lane urban road during daytime.

the right lane (approx 50m). Left lane is clear. 
Traffic light ahead shows green. Pedestrian on right 
sidewalk, not crossing. No immediate hazards detected.

Generation took ~X seconds for ~80 tokens. Not real-time, but useful for logging, annotation, or second-opinion systems.

Lessons Learned

VLMs are Memory-Bound

Unlike vision models where compute is the bottleneck, VLM inference on Jetson is entirely memory-bound. Quantization is mandatory, not optional.

Vision Encoder Matters

The vision encoder (often ViT-based) needs memory too. Qwen2.5-VL's dynamic resolution handling is elegant but increases memory for large images. Resize inputs to 384×384 or smaller.

TTFT Dominates User Experience

For interactive applications, time-to-first-token matters more than tokens/sec. Users perceive 2-second TTFT as "slow" even if throughput is decent.

Conclusion

  • Sub-4B VLMs fit on Orin Nano with INT4/INT8 quantization
  • ~6GB is the practical memory budget with desktop running
  • llama.cpp with GGUF is the simplest deployment path
  • Driving scene understanding works but isn't real-time

The Jetson Orin Nano isn't an LLM powerhouse, but it can run useful VLM inference for edge applications where cloud connectivity is limited or latency matters.
