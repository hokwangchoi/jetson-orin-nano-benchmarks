Three runtimes, one platform bug: a VLM on Jetson Orin Nano

Hokwang Choi · April 2026 · 15 min read

TL;DR

I set out to benchmark Cosmos-Reason2-2B, NVIDIA's physical-AI VLM, across three inference runtimes on a Jetson Orin Nano 8 GB: llama.cpp, vLLM, and TensorRT Edge-LLM.

What actually happened: vLLM and TRT-Edge-LLM both got blocked by the same JetPack 6.2.1 memory-allocation bug, in two different ways. NVIDIA shipped a partial fix (JetPack 6.2.2 / L4T 36.5) mid-work that unblocked one of them with some config tweaks; the other needed a kernel-level CMA change plus an ONNX graph rewrite to get past. As far as I can tell this is the first public report of Cosmos-Reason2-2B running on TRT-Edge-LLM on Orin Nano — NVIDIA's own supported matrix pairs this combination with Thor, not Orin.

Final numbers: TRT Edge-LLM 60 TPS, vLLM 56 TPS, llama.cpp 38 TPS on Cosmos-Reason2-2B at W4A16 / Q4_K_M respectively. TRT wins decode throughput and uses the least memory (4.3 GB peak). vLLM wins image first-token latency when warm (77 ms vs TRT's 420 ms). llama.cpp wins context length (4096 vs 1024) and has a cleaner startup story.

Code: github.com/hokwangchoi/jetson-orin-nano-benchmarks

1. Why this comparison

VLMs are showing up in robotics and autonomous-vehicle stacks as the "brain" that turns a scene into grounded reasoning. The interesting models are small now — 2B to 4B parameters, multimodal, post-trained for physical reasoning. The interesting hardware is cheap edge silicon — Jetson Orin Nano is $250, 8 GB unified memory, 67 TOPS.

I wanted to know what the actual cost of moving one of these models onto one of these boards looks like in 2026. Not TOPS-on-paper, not Hopper numbers — just: set up the hardware, pick a model, run it, measure.

2. The hardware budget

8 GB of unified memory is the hard constraint. Your weights, your KV cache, your vision encoder activations, and the Linux desktop all share the same physical DRAM. Every byte matters.

3. The model: Cosmos-Reason2-2B

NVIDIA's Cosmos-Reason2-2B is a Qwen3-VL-2B architecture post-trained with SFT and RL on embodied-reasoning datasets. It's the right size for an 8 GB board when quantized, and it outputs chain-of-thought in an explicit <think>...</think><answer>...</answer> format which makes evaluation tractable.

It's also the model NVIDIA's own VLM tutorials point at for Jetson. The 2B variant is the one that fits; the 8B is for AGX Thor and above.

4. The original plan

Three C++-native runtimes on the same model, same hardware, same prompts.

The angle is "what's it like to deploy a production runtime on an edge platform in 2026." Not "which is fastest" — that question is boring when one of them won't start.

5. Rabbit hole 1: vLLM won't initialize

First attempt. Pull the NVIDIA-AI-IOT vLLM container, load the model, watch PyTorch crash on startup: 

NvMapMemAllocInternalTagged: 1075072515 error 12
NvMapMemHandleAlloc: error 0
Error : NVML_SUCCESS == r INTERNAL ASSERT FAILED at
  "/opt/pytorch/pytorch/c10/cuda/CUDACachingAllocator.cpp":838

That first line is the kernel driver rejecting a GPU memory allocation. PyTorch's CUDA caching allocator queries NVML to plan its memory budget; when NVML's free-memory answer disagrees with what the driver will actually hand out, PyTorch asserts and dies.

Searching the error turns up a large and very active NVIDIA forum thread. A moderator acknowledged this in November 2025: "This issue is related to the r36.4.7 update and has a failure rate." The underlying cause is a security patch (CVE-2025-33182) that changed how NvMap — the Tegra-specific memory subsystem — accounts contiguous CUDA allocations. A lot of workloads that used to work now fail.

At this point I had two paths: invest in debugging vLLM's allocator interaction, or pivot to a comparison that actually runs. I pivoted.

6. Pivot: llama.cpp + TRT Edge-LLM

llama.cpp doesn't use PyTorch's allocator or query NVML. It uses raw cudaMalloc in small chunks as layers execute. That allocation pattern sidesteps the bug. Within a few hours it was serving.

llama.cpp: what worked

Concrete setup:

# On the Jetson, via NVIDIA's official llama.cpp container
# Model: robertzty/Cosmos-Reason2-2B-GGUF (BF16 split + mmproj)
# Merge splits, quantize BF16 → Q4_K_M (~1.1 GB)

sudo docker run --rm --runtime=nvidia --network host \
    -v ~/models/cosmos-reason2-2b:/models:ro \
    ghcr.io/nvidia-ai-iot/llama_cpp:latest-jetson-orin \
    llama-server \
        --model   /models/Cosmos-Reason2-2B-Q4_K_M.gguf \
        --mmproj  /models/mmproj-Cosmos-Reason2-2B-BF16.gguf \
        --ctx-size 4096 \
        --n-gpu-layers 99 \
        --parallel 1 \
        --flash-attn on \
        --jinja

Headless boot, sudo nvpmodel -m 2 && sudo jetson_clocks for MAXN_SUPER, 11 GB swap as a safety net. Serving on port 8000 in ~8 seconds from cold start (mmap-friendly GGUF).

7. Rabbit hole 2: TRT Edge-LLM hits the same wall, from a different angle

TRT Edge-LLM is NVIDIA's C++-native inference runtime for embedded platforms. The workflow is:

HuggingFace checkpoint │ ▼ ┌─────────────────────────┐ │ quantize-llm (on host) │ INT4 AWQ, ModelOpt, ~4 min on A40 └───────────┬─────────────┘ ▼ ┌─────────────────────────┐ │ export-llm (on host) │ → ONNX + external weights │ export-visual (on host) │ → ViT encoder ONNX └───────────┬─────────────┘ ▼ (~2 GB scp to Jetson) ┌─────────────────────────┐ │ llm_build (Jetson) │ ONNX → TRT engine (THIS IS WHERE IT BREAKS) │ visual_build (Jetson) │ hardware-specific tactics └───────────┬─────────────┘ ▼ ┌─────────────────────────┐ │ llm_inference (Jetson) │ JSON input, CUDA-graph-captured execution └─────────────────────────┘

Host-side quantize and ONNX export ran fine on an A40 pod. Runtime build on the Jetson took ~40 minutes but succeeded. Engine build is where things died: 

[INFO] Timing Runner: {ForeignNode[/Unsqueeze.../Cast]} (Myelin[0x80000023])
[INFO] MemUsageChange: CPU 2380 MB, GPU 3710 MB
NvMapMemAllocInternalTagged: 1075072515 error 12   (12 retries)
[ERROR] CUDA error 2 for 622329856-byte allocation
[ERROR] Could not find any implementation for node {ForeignNode[/Unsqueeze.../Cast]}.
[ERROR] Failed to build LLM engine.

Same kernel driver, same error signature, different codepath. This isn't PyTorch's NVML assertion — TRT uses raw cudaMalloc through its autotuner. The specific request that fails is exactly 1,075,072,515 bytes (~1 GiB) for a fused "Myelin" subgraph. I originally identified this as the rotary position embedding path; on later re-investigation it turned out to be the LM head output projection — /lm_head/MatMul (2048 × 151936) fused with the fp16→fp32 Cast for logits output. The shape TRT reports just before the failure (Float(151936, 151936, 1)) gives it away: 151936 is the Qwen3 vocab size, and the fallback 622 MB tactic is exactly the fp16 LM head weight matrix. Getting the node identity right mattered later because it redirected the fix: shrinking sequence length can't help, but rewriting the LM head graph can. TRT retries tactic 0 twelve times, then falls back to asking for 622 MB — which also fails because by then the memory context is wedged.

What I tried, what didn't work

  • Reduce --maxInputLen / --maxKVCacheCapacity. The kernel-level allocation cap is contiguous-size, not total-memory; shrinking the model's working set doesn't help if a single fused kernel still needs 1 GB of scratch space for tactic selection.
  • Patch builderUtils.cpp to set kWORKSPACE pool limit. Accepted by the TensorRT API, ignored by Myelin — the pool limit binds a different codepath.
  • Patch to set kTACTIC_DRAM limit. Tried capping at 2 GiB, then 768 MiB (below tactic 0's 1 GiB request). Neither bound the failing allocation — Myelin's autotuner pool is separate from anything IBuilderConfig exposes on TRT 10.3.
  • Re-export with --trt_native_ops, which decomposes the Edge-LLM custom attention plugin into stock ONNX ops. The resulting ONNX has an Attention op that TensorRT 10.3's ONNX parser can't consume. Dead end.
  • Free as much memory as possible before build (headless boot, stop docker/snapd/jtop, drop caches). Got to 6.9 GB available. Build still failed at the same allocation — confirming the issue is contiguous-size, not total-memory.

At this point the diagnosis is clear: the Myelin subgraph in TRT 10.3 wants one specific 1 GB contiguous allocation, and L4T 36.4.7's NvMap refuses it. Same underlying bug as vLLM's failure, surfacing in a completely different codepath.

8. The first breakthrough: NVIDIA ships a partial fix

Mid-investigation, NVIDIA released JetPack 6.2.2 with L4T 36.5.0. The release notes explicitly call out a critical CUDA memory allocation problem introduced in 6.2.1 and describe it as the highest-priority fix in the release.

Upgrade path is apt, not a reflash:

# Edit /etc/apt/sources.list.d/nvidia-l4t-apt-source.list:
#   r36.4 → r36.5 in all deb lines
sudo apt update
sudo apt dist-upgrade -y
sudo reboot

Kernel jumps from 5.15.148-tegra to 5.15.185-tegra. CUDA stack rebuilds initramfs, bootloader slot swaps, first boot takes ~3 minutes.

What the fix does and doesn't

The fix is partial. L4T 36.5 resolves the tiny-allocation NVML path, so PyTorch can initialize. But loading a 5 GB BF16 model still triggers a variant of the same bug at the multimodal profiling stage — the ViT positional-embedding path asks for a contiguous 1 GB allocation and gets the same rejection. TRT-Edge-LLM's Myelin autotuner hits the same wall out of the box. §9 and §10 describe how both got past it.

9. After the fix: vLLM with W4A16

Two shifts were required to get vLLM actually serving:

  1. Model swap. The raw nvidia/Cosmos-Reason2-2B at BF16 is ~5 GB of weights plus PyTorch allocator overhead — it loads, but dies at multimodal profiling where the encoder cache allocates on top of loaded weights. I switched to embedl/Cosmos-Reason2-2B-W4A16, a community-quantized W4A16 AWQ version that takes 2.29 GB of GPU memory. Embedl is a small edge-AI company in Sweden; their port sidesteps the problem by shrinking the weight footprint enough that the rest of the pipeline has room to breathe.
  2. Cap the vision encoder profile. vLLM's startup profile runs the ViT at its declared maximum image size to size the encoder cache. On Qwen3-VL that default is ~1.4 M pixels, which asks the allocator for ~1 GB contiguous and cascades into the same NvMap failure mode from §8. The fix is to pass --mm-processor-kwargs '{"num_frames":2,"max_pixels":150528}' and --limit-mm-per-prompt '{"image":1,"video":1}'. Neither flag appears on Embedl's model card; without them the server won't start on an 8 GB board.

Final working command:

sudo docker run --rm --runtime=nvidia --network host \
    --shm-size=8g --ulimit memlock=-1 --ulimit stack=67108864 \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    -e HF_HOME=/root/.cache/huggingface \
    ghcr.io/nvidia-ai-iot/vllm:latest-jetson-orin \
    vllm serve embedl/Cosmos-Reason2-2B-W4A16 \
        --host 0.0.0.0 --port 8000 \
        --max-model-len 1024 \
        --gpu-memory-utilization 0.65 \
        --max-num-seqs 1 \
        --mm-processor-kwargs '{"num_frames":2,"max_pixels":150528}' \
        --limit-mm-per-prompt '{"image":1,"video":1}'

A few things to know if you're reproducing this. Run headless. --gpu-memory-utilization 0.65 on an 8 GB board needs the desktop killed (sudo systemctl isolate multi-user.target) and swap off (sudo swapoff -a); with the desktop up, vLLM refuses to start because there isn't enough free memory to hit the requested utilization. Context length trades against image headroom. The Qwen3-VL visual tokenizer emits about 770 tokens per image at the 150528-pixel cap; at max-model-len 1024 that leaves ~240 tokens for the prompt and generation, which is enough for short outputs but not long reasoning traces. If you need more decode budget, you can shorten max_pixels further (fewer visual tokens per image) or accept that long generations are text-only. Don't pass --enforce-eager. vLLM 0.19's defaults already enable CUDA graphs, chunked prefill, and prefix caching; the startup log confirms Capturing CUDA graphs (decode, FULL): 1/1 and Using MarlinLinearKernel for CompressedTensorsWNA16. Some community recipes add --enforce-eager as a memory workaround, which disables the graphs and costs roughly 3× in decode throughput. On an 8 GB board the memory savings aren't worth it — the graphs only take ~10 MB on this model.

Five runs each, streaming measurement, temperature 0. Text prompt is 30 tokens generating 128; image prompt is 20 tokens over bus.jpg generating up to 128 (the model stops at its EOS around 101):

Medians across 5 streaming runs, temperature 0. Variance is tight — TPOT holds to ±0.1 ms once warm. One thing worth flagging: the first image request in any batch runs ~500 ms–3 s for TTFT because vLLM does not capture CUDA graphs for the vision encoder (cudagraph_mm_encoder: False). The ViT path compiles lazily on first use per unique input shape. After one or two warmup image requests, TTFT settles to ~77 ms. If your deployment is a stream of similarly-shaped frames, this is a one-time cost; if every frame changes resolution, you'll pay it every time. An alternative is to pre-warm the encoder at serve startup with a dummy image at the same resolution as your production stream.

Memory pressure while serving

The board sits close to full while vLLM serves:

$ free -h
               total        used        free      shared  buff/cache   available
Mem:           7.4Gi       6.9Gi        92Mi        10Mi       492Mi       362Mi
Swap:             0B          0B          0B

6.9 GB resident with 362 MB available, swap off. vLLM accounts for ~5 GB (weights + KV cache sized for 1024 context + CUDA graph pools + activations + CUDA context). The remaining 1.9 GB is the OS, docker, and the usual Linux overhead. This is the edge of what Orin Nano 8 GB + vLLM + a 2B-class VLM allows. Push either --gpu-memory-utilization or --max-model-len much further and the server starts refusing to allocate — in my testing, max-model-len 2048 was already past the line.

Iteration cost: NvMap leaks between failed runs

Sitting this close to the memory ceiling means every config tweak is a coin flip, and a lot of them fail. On Tegra, failing is expensive: a crashed vLLM process leaves NvMap mappings that docker rm -f does not fully reclaim. After a startup failure, CUDA reports several hundred megabytes less free memory than free -h shows at the system level. The next attempt can then OOM at a --gpu-memory-utilization value that would otherwise succeed — which makes the leak look like a bad config.

sudo sysctl -w vm.drop_caches=3 && sudo docker system prune -f recovers most of it; only a reboot recovers all of it. In practice, iterating on vLLM config on an Orin Nano benefits from a reboot every few failed attempts. Not something you'd notice on a GPU workstation, where the driver cleanly releases memory between runs. This board is closer to an embedded device than a server, and the tooling around it hasn't caught up to that framing.

Head-to-head: Cosmos-Reason2-2B across runtimes

Runtime Quantization Context TTFT text TTFT image TPOT TPS Peak mem

Same model, same quantization class (INT4 weights, FP16 activations), same hardware. Three distinct winners depending on what you optimize for. TRT leads on decode (60 vs 56 TPS), on text TTFT (29 ms, 2× faster than either alternative), and on peak memory (4.3 GB vs vLLM's 6.9 GB — a 37% reduction). vLLM wins image TTFT post-warmup (77 ms vs TRT's 420 ms) because its vision encoder gets lazily compiled into CUDA graphs on first use and cached for subsequent requests. TRT pays the full ~200 ms ViT cost every time. llama.cpp loses on speed but wins context length (4096 vs 1024) — the only runtime on this hardware that can handle long reasoning traces. The TRT row took the most work to get: see §10 for what it took.

The TPOT numbers deserve a closer look. llama.cpp, vLLM and TRT are all memory-bandwidth-bound on a 2B model at batch 1 — Orin Nano's 68 GB/s LPDDR5 caps everyone — so the decode gap is small in absolute terms (10 ms between fastest and slowest). Where the numbers separate more is TTFT: llama.cpp and vLLM both fuse prefill aggressively but use very different kernel paths, and TRT's pre-compiled engine is just dense TensorRT tactics on the hot path. The 29 ms text TTFT is pure prefill time on a short prompt — no startup overhead, no tokenization surprise. That number is what vLLM approaches for cached prompts and llama.cpp approaches for the first generate call after a reuse hit.

Why vLLM wins decode over llama.cpp (but loses to TRT): the W4A16 AWQ weights dispatch through vLLM's Marlin GEMM kernels (Using MarlinLinearKernel for CompressedTensorsWNA16), which are tuned for Ampere tensor cores and do on-the-fly INT4→FP16 dequantization in the GEMM prologue. llama.cpp's Q4_K_M GGUF goes through ggml-cuda kernels that are excellent but more general-purpose. TRT uses its own Int4GroupwiseGemmPlugin (PluginV3), which is one of a small number of hand-tuned plugins that ship with Edge-LLM — roughly the same kernel family as Marlin but wired directly into a fused CUDA graph with no Python in the path. That's where the ~3.5 ms TPOT advantage over vLLM comes from.

Why llama.cpp wins context length: vLLM pre-allocates the KV cache upfront at max-model-len, multiplied by the number of layers times the number of KV heads. On a 2B model at 8 GB with 5 GB already spoken for by weights, graphs, and vision encoder activations, 1024 is the largest context that still passes the startup sanity check. TRT Edge-LLM has the same structural limit — max KV cache capacity is baked into the engine at build time. llama.cpp allocates KV cache on demand per active slot rather than reserving the full max-context budget upfront, and uses mmap for weights so the OS can swap cold weight pages to disk. 4096 context fits without drama. This is a real functional difference, not a benchmark artifact: if your deployment needs long reasoning traces, llama.cpp is the only option on this hardware today.

10. Past the second wall: three interventions to get TRT Edge-LLM building

The TRT blocker from §7 stayed closed for a while. The notes I wrote at the time concluded the Myelin autotuner behavior was a fixed property of TRT 10.3 on L4T 36.5 and the fix would need to come from upstream. That turned out to be half right — no combination of TRT builder flags or source patches fixes it from inside Edge-LLM. But a kernel-level CMA change plus an ONNX graph rewrite bypass the problem cleanly. Here's what that took.

Correctly identifying the failing node

The error message reports "{ForeignNode[/Unsqueeze.../Cast]}", which is TRT shorthand for a fused subgraph whose first op is an Unsqueeze and whose last op is a Cast. I originally guessed this was the rotary position embedding math. It isn't — it's the LM head. Tracing the actual ONNX graph, the fused region runs from /GatherND (picks the last-token hidden state) through /lm_head/MatMul (the 2048 × 151936 projection to vocab logits) to /Cast (fp16 → fp32 for the logits output).

The allocation sizes confirm it. Tactic 0's 1,075,072,515 bytes is roughly 1.5 × 151936 × 2048 × 2 — Myelin's workspace for the fused kernel. The 622,329,856-byte fallback allocation is exactly 2 × 2048 × 151936, the fp16 LM head weight matrix. The embedding tensor in the ONNX export on disk is 622,329,944 bytes, within a few bytes of the same number. It's the LM head, fused with the logits cast.

Getting the node identity right mattered because it told me which interventions could and couldn't help. Shrinking --maxInputLen can't help: the allocation is vocab × hidden, not seq_len × hidden. Neither can reducing batch size. But rewriting the LM head to break the fusion boundary can, because then Myelin has two smaller candidates to pick from instead of one big one.

Intervention 1: enlarge the CMA pool

Default Jetson CMA (the kernel's Contiguous Memory Allocator) is 256 MiB. Myelin's tactic 0 wants 1,075 MiB contiguous; it falls back to a 593 MiB tactic if tactic 0 fails cleanly. With 256 MiB CMA, neither fits, and tactic 0's 12 failed retries progressively exhaust the pool so that by the time the fallback runs, no contiguous memory is free. The behavior I actually observed matched this exactly: once I started watching CmaFree in /proc/meminfo during a build, I saw the pool drain to 0 during tactic 0's retry loop.

Setting cma=950M on the kernel command line (via /boot/extlinux/extlinux.conf) raises the pool to 960 MiB. Why 950M and not 2G: Jetson's low physical memory comes in two ~992 MiB chunks separated by a 64 MiB firmware carveout, and the ARM64 kernel hardcodes CMA placement below 4 GiB — cma=2G@0x100000000-0x280000000 is rejected with "exceeds limit (0x100000000)". 950M is effectively the ceiling on this hardware — the largest value that fits in one of the 992 MiB low-memory chunks with some slack.

With 960 MiB CMA, tactic 0 still exceeds the pool (1025 > 960), but it now fails on a fast size check rather than wedging the pool through partial allocations. The CMA trace makes the difference visible. Failing run, default 256 MiB CMA: 

19:59:09  CmaFree 779 MB   ← idle, just before Myelin starts the LM head
19:59:13  CmaFree  36 MB   ← tactic 0 retry 4 or so
19:59:14  CmaFree   0 MB   ← pool wedged
19:59:18  CmaFree   0 MB
19:59:21  CmaFree 932 MB   ← released only after the build error propagates

Succeeding run, 960 MiB CMA plus the surgery below: CmaFree briefly drops to ~350 MB during tactic 1's 593 MB allocation and then cleanly releases. No wedging, no retry loop.

Intervention 2: ONNX graph surgery to split the LM head

CMA expansion alone isn't enough — tactic 0 gets attempted first and its 1025 MB request still fails. The clean fix is to stop Myelin from ever trying tactic 0, by preventing it from fusing the LM head into a single node that needs such a large scratch space.

The surgery splits the 2048 × 151936 LM head MatMul along the vocab dimension into two 2048 × 75968 halves, then stitches the outputs back together with a Concat(axis=-1). Two smaller fusion candidates each well under 600 MiB tactic scratch. Output is mathematically identical — this is a graph rewrite, not a model change. 

Before:                             After:
                                                    ┌─ MatMul_lo (2048 × 75968) ─┐
... → GatherND → MatMul → Cast →    ... → GatherND ─┤                            ├─ Concat → Cast →
              (2048×151936)                         └─ MatMul_hi (2048 × 75968) ─┘

First attempt used Identity nodes as fusion barriers. That didn't work: TRT's graph optimizer constant-folds and removes Identity nodes before Myelin runs its pattern matcher. Concat is a real data-rearranging op and the optimizer leaves it in place. With the split in place the build completes in about ten minutes. The script is split_lm_head.py in the repo.

Intervention 3 (documentary): patching kTACTIC_DRAM

I also tried a one-line patch to cpp/builder/builderUtils.cpp changing setMemoryPoolLimit(kTACTIC_DRAM, 2ULL << 30) to 768ULL << 20 — the theory being that a 768 MiB cap below tactic 0's 1 GiB request would make Myelin skip it. It doesn't. The kTACTIC_DRAM pool binds a different codepath; Myelin's autotuner has its own scratch pool that IBuilderConfig does not reach on TRT 10.3. I left the patch in the repo as documentary record of what was tried, with a comment in the source. Future TRT versions may wire this pool through, at which point the patch would start being useful and could replace the graph surgery.

Cost of the workaround

The split introduces one extra kernel launch (the Concat) per forward pass. At batch 1 this is microseconds — not visible in the benchmark numbers. The engine runs clean through llm_inference, produces a fluent description of the bus image ("two individuals walking side by side on a paved sidewalk... a blue bus with a white roof and a green logo parked on the street..."), and benchmarks at 60 TPS decode — matching vLLM within the memory-bandwidth ceiling on this hardware.

The full diagnostic writeup — including what didn't work (reducing seq_len, the kWORKSPACE patch, --trt_native_ops re-export, Identity barriers) — is in vlm-benchmarks/notes/trt_edgellm_cosmos_resolution.md.

11. What I actually learned

Production runtimes are platform-fragile at the edge

On a desktop with an RTX 4090, you pick a runtime, install it, and it works. On a Jetson running a recently-patched L4T, the same runtime may or may not start — depending on whether its allocator happens to step on a specific accounting bug in a security patch. The "works on my machine" surface area between desktop and edge is large, and you don't discover it until you try to ship.

Same bug, different codepaths, different symptoms

The L4T 36.4.7 NvMap issue surfaced in three distinct ways on three runtimes: a PyTorch NVML assertion at startup (fixed in 36.5), a ViT multimodal-profiling failure after model load (still present for BF16 models in 36.5), and a TRT Myelin autotuner failure during engine build (still present in 36.5). llama.cpp's small-chunk allocation pattern happens to sidestep all three. Knowing how your runtime allocates memory is suddenly a first-class deployment concern, not a trivia question.

Community ports carry real weight on constrained platforms

NVIDIA's official Cosmos-2B vLLM recipe for Orin Nano uses their own FP8 checkpoint from NGC. I ended up using Embedl's W4A16 from HuggingFace instead — equivalent story (pre-quantized, fits in 8 GB), but more aggressively compressed and a simpler download. On memory-constrained hardware, the right model isn't always the original checkpoint; it's a community-optimized variant that was built knowing the deployment target.

Diagnose before you patch

I spent an hour writing cudaMalloc probes before applying any fixes. The probes told me the bug was not a simple size cap — raw allocations up to 1 GB succeeded standalone — which saved me from several hours of patching the wrong thing. Cheap probes before long builds is a good rule.

Portability vs. optimality, made concrete

llama.cpp ships pre-compiled generic CUDA kernels that run on any SM and allocates memory in small chunks. TRT Edge-LLM benchmarks kernel variants on your specific silicon at build time and asks the driver for large contiguous scratch buffers. llama.cpp's approach is why it kept running when the NvMap bug surfaced — it never triggered it. TRT's approach is what you want in production when the build succeeds: a fully-fused engine with predictable 17 ms per-token latency and a 4.3 GB memory footprint, 37% less than vLLM on the same model. The cost of that predictability is the 10-minute build, the CMA kernel parameter, and the ONNX surgery from §10. Fair trade if the deployment is stable; painful if you're iterating on model variants.

vLLM sits in between: a general-purpose Python + PyTorch serving layer with specialized kernels (Marlin for W4A16, FlashAttention for prefill, CUDA graphs for decode) wired in for the hot paths. That mix is why it beats llama.cpp's more uniform ggml-cuda kernels on decode throughput for this specific quantization, while still running without a per-device build step. The price is a bigger steady-state memory footprint and a more fragile startup path — the same reasons the NvMap bug hit it harder than it hit llama.cpp. Three different answers to the same portability-vs-optimality tradeoff, and on this hardware the answer that wins depends on which metric you're optimizing for.

Vendor support matrices are lower bounds, not ceilings

NVIDIA's Jetson AI Lab tutorial pairs Cosmos-Reason2 8B with Thor (128 GB unified memory) and pairs Orin Nano with Qwen3-4B-Instruct — a text-only LLM, not a VLM. There's no official recipe anywhere for Cosmos-Reason2-2B on Orin Nano via TRT-Edge-LLM. That absence is informative: it tells you NVIDIA's engineers found the combination unpleasant enough to route it out of their tutorials. It isn't a ban, though. With a kernel-level CMA change and an ONNX graph rewrite, the combination works and performs competitively. "Not officially supported" on an edge platform often means "works after you do the platform-level work to make it work" — not "actually impossible."

Watch the allocator, not the model

Two of the three bugs I hit on this project were allocator bugs, not model bugs. vLLM's startup failure was a CUDA caching allocator assertion caused by NVML disagreeing with the kernel driver. TRT Edge-LLM's build failure was the kernel's CMA pool exhausting under Myelin's retry behavior. Both looked like model problems at first glance — "the VLM doesn't fit" — but the root causes were downstream of anything you'd find by reading the model architecture. On memory-constrained edge hardware, live-instrumenting the allocator (/proc/meminfo, CmaFree, per-process NvMap tracking) is more often the right move than reading more TRT docs.
